# pip — v1 design

**Status:** Accepted — implemented in commits 38bfdc3..04c9a98
**Date:** 2026-04-07
**Author:** mattwhite (with Claude)

## Overview

pip is an open-source drop-in help widget that any website can add in essentially one line of code. A small mouse icon floats in the corner of the host site; users click it, ask a question in natural language, and get a contextual answer grounded in the site's own docs. The mouse can also point — visually — at the specific element the user should interact with next.

The combination of *contextual to this specific page* + *visual guidance* + *drop-in via script tag* is the differentiator. Existing options are either dumb (static FAQs), expensive SaaS (Intercom, Zendesk), or context-free (a ChatGPT iframe that knows nothing about the host app).

## Goals

- **One-line integration** for the common case. A single `<script>` tag (or a single `import`) should get a working widget on the page.
- **Self-hostable and fully open source.** Devs own their LLM keys, their endpoint, and their data. No mandatory SaaS.
- **Genuinely useful out of the box.** Default behavior (screenshot + a markdown context file + Claude) should feel smart, not like a generic chatbot.
- **Privacy-defensible.** Client-side redaction, first-use consent, and a persistent kill switch — good enough that a security-conscious dev can actually ship it.
- **Maintainable by one or two people.** Small surface area, few dependencies, boring tech.

## Non-goals (v1)

- Full agent actions (clicking, typing, navigating on the user's behalf). Deferred to v2.
- Vector search over crawled docs. Use the `getContext()` hook instead; devs can plug in whatever retrieval they want.
- Persistent chat history across sessions. Ephemeral only.
- Hosted SaaS backend. Self-host only for v1.
- Framework-specific bindings (React/Vue/Svelte components). The vanilla widget self-mounts and works in all of them.
- Voice input, multi-language UI, analytics dashboards.

## V1 scope

The minimum shippable product:

1. **Chat UI** — a small mouse icon floats in the corner. Click opens a multi-turn chat panel with ephemeral in-memory history.
2. **Contextual answers** — each turn sends the user's question, recent chat history, a screenshot of the current page, URL + page title, a redacted DOM snapshot, and a dev-supplied markdown context blob to an LLM.
3. **Guided pointing** — the LLM can call a `highlight` tool with pixel coordinates + a short caption. The widget draws a dimmed backdrop, freezes scroll, and renders an SVG arrow at the coordinates. Overlay auto-dismisses on scroll, resize, or click.
4. **Privacy rails** — password fields auto-redacted; dev-configurable redaction selectors; `[data-pip="redact"]` attribute; first-use consent dialog; persistent kill switch in the widget chrome.
5. **Self-host adapter** — `@pip-help/server` package with a Next.js App Router route handler and a generic fetch-style handler that wraps the Vercel AI SDK.

### Key design choices

| Decision | Choice | Why |
|---|---|---|
| MVP scope | Chat + guided pointing (no agent actions) | Proves the core magic without the trust/safety cliff of autonomous action-taking. |
| Hosting model | Self-host + hosted path optional later | Clerk/Supabase/PostHog playbook. OSS-first, no infra debt in v1. |
| Context source | Dev-supplied markdown, pluggable `getContext()` | Honest "one line of code"-adjacent. Power users can plug vector search. |
| Pointing mechanism | Coordinate-based with scroll freeze | Simplest path to a working demo. Upgrade to Set-of-Mark (numbered overlays) in v2. |
| Distribution | Script tag + NPM, vanilla TS core | One codebase, two channels. Shadow DOM keeps the bundle framework-free. |
| Privacy | Client-side redaction + consent + kill switch | Redaction happens before the fetch — dev's backend never sees sensitive data. |
| LLM integration | Vercel AI SDK v6 + AI Gateway | Provider-agnostic by default; `ToolLoopAgent` handles tool-calling loop; `toUIMessageStreamResponse()` handles streaming + vision + tools. |
| Chat mode | Multi-turn, ephemeral in-memory | Modern UX with zero persistence infra. History cleared on reload. |
| Default model | `anthropic/claude-sonnet-4.6` (via AI Gateway) | Vision-capable, fast, good price/quality for this task. Gateway model string — swap providers without changing code. |

## Component design

### Client — `@pip-help/core`

Pure vanilla TypeScript, no framework runtime. Shadow DOM isolation so host site CSS can't break the widget and vice versa.

- **`src/index.ts`** — public API and auto-mount. Exports `pip.mount(config)`; script-tag users get auto-mount via the `data-pip-endpoint` attribute.
- **`src/widget/`** — UI components (vanilla web components).
  - `mouse-button.ts` — floating mouse icon, opens the chat panel.
  - `chat-panel.ts` — scrollable chat history, input box, send button, pause toggle.
  - `consent-dialog.ts` — first-use consent UI, writes to localStorage.
- **`src/capture/`** — page state capture.
  - `screenshot.ts` — takes a viewport screenshot via `html-to-image` (or `modern-screenshot` — decide in impl).
  - `dom-snapshot.ts` — walks the DOM and emits a trimmed representation of interactive elements + their visible text.
  - `redact.ts` — auto-redacts `input[type=password]`, `[data-pip="redact"]`, and any selectors in `config.redact`. Blacks out bounding boxes in the screenshot *and* strips text from the DOM snapshot.
- **`src/overlay/`** — pointing arrow renderer.
  - `arrow.ts` — SVG arrow pointing at `(x, y)` with a caption.
  - `freeze.ts` — dimmed backdrop, scroll lock, auto-dismiss on scroll/resize/click.
- **`src/transport/`** — fetch client.
  - `stream.ts` — reads the streaming response, parses AI SDK data stream protocol, dispatches text deltas and tool calls to the chat panel and overlay.
- **`styles.css`** — scoped to the Shadow DOM. Minimal CSS. Dark/light auto via `prefers-color-scheme`.

### Server — `@pip-help/server`

Thin wrapper over the Vercel AI SDK. Framework-agnostic core plus thin adapters.

- **`src/handler.ts`** — framework-agnostic `createPipHandler(config)`. Takes a standard `Request`, constructs a `ToolLoopAgent`, and returns `agent.respond(...).toUIMessageStreamResponse()` for the client to consume.
- **`src/next.ts`** — re-exports `createPipHandler` as a Next.js App Router route handler (typed `POST`).
- **`src/agent.ts`** — builds the `ToolLoopAgent` (AI SDK v6 pattern): system prompt + context markdown + the `highlight` tool. Typed so `InferAgentUIMessage<typeof agent>` is exportable for the client parser.
- **`src/prompt.ts`** — builds the system prompt: static instructions + the dev's context markdown + chat history. Enforces a token budget by trimming old turns.
- **`src/tools.ts`** — defines the `highlight` tool using AI SDK v6's tool API with a Zod `inputSchema` (`{ x, y, width, height, caption }`). Read-only — no `execute` in v1 (the tool call is interpreted client-side by the overlay renderer).
- **`src/types.ts`** — shared request/response types + the exported `PipUIMessage` type (from `InferAgentUIMessage<typeof agent>`). Consumed by `@pip-help/core` at build time for end-to-end type safety.
- **`src/context.ts`** — default `getContext` implementation: reads a markdown file path. Devs pass their own for custom retrieval.

## Data flow (one chat turn)

1. User clicks the mouse button → chat panel opens (Shadow DOM).
2. User types a question and hits Enter.
3. `capture/` takes a viewport screenshot via `html-to-image`.
4. `capture/` walks the DOM, strips password inputs and dev-flagged selectors, blacks out their bounding rects on the screenshot, and emits a trimmed DOM snapshot.
5. `transport/` POSTs `{ messages, screenshot, url, title, redactedDom, viewport }` to the dev's configured endpoint and opens a stream.
6. Dev's handler calls `getContext({ query, url, pageState })` (default: read a markdown file).
7. Handler builds the `ToolLoopAgent` with the system prompt + context markdown + the `highlight` tool, then calls `agent.respond({ messages })` where messages include prior chat history + the current user turn (screenshot attached as a vision part).
8. LLM streams text. If it decides to point at something, it emits a `highlight` tool call with `{ x, y, width, height, caption }`.
9. Handler returns `.toUIMessageStreamResponse()` so the client receives the standard AI SDK UI message stream (text deltas + tool-call events + finish markers).
10. Widget parses the UI message stream directly (no `@ai-sdk/react` — we're vanilla TS). Text deltas render into the chat bubble; `highlight` tool-call events trigger `overlay/` to freeze scroll, draw the dimmed backdrop + SVG arrow, and show the caption.
11. Overlay dismisses on: user click anywhere, scroll, resize, or a follow-up question.

## Privacy design

- **Password inputs auto-redacted**, always, no config. Screenshot is edited client-side to black out the bounding rect of every `input[type=password]`.
- **Dev-supplied selectors** via `config.redact: string[]`. Matching elements get the same treatment.
- **Opt-in attribute** — `[data-pip="redact"]` on any element.
- **Custom hook** — `config.beforeSend: (payload) => payload` lets power users transform the entire payload before it leaves the browser.
- **First-use consent** — first open in a session shows a small dialog: "pip will share a screenshot of this page with an AI assistant to help you." Consent persists in `localStorage` under `pip-consent`.
- **Kill switch** — persistent "pause pip" toggle in the widget chrome. Disables the session entirely until re-enabled.
- **Debug mode** — `config.debug: true` logs the outgoing payload to the console and skips the network request. Documented in the README.

Critically, **all redaction happens client-side before the fetch**. The dev's backend never receives sensitive data. This is safer than most analytics tools in the same category.

## Tech stack

- **Language:** TypeScript 5.x, strict mode.
- **Client bundler:** tsup (esbuild-based; outputs IIFE + ESM from one config).
- **Client runtime:** vanilla TS. Web Components + Shadow DOM. No framework dependency.
- **Screenshot:** `html-to-image` or `modern-screenshot` (evaluate both in implementation). Target is a small dependency that handles the common DOM case; cross-origin images are acceptable as a known limitation.
- **Server SDK:** Vercel AI SDK v6 (`ai` package). Uses the `ToolLoopAgent` pattern with Gateway model strings (`provider/model`) by default, so devs don't need provider-specific packages. Direct provider SDKs (`@ai-sdk/anthropic`, etc.) supported for self-hosted or non-Gateway setups.
- **Default model:** `anthropic/claude-sonnet-4.6` via AI Gateway.
- **Exact API signatures** will be verified against `node_modules/ai/docs/` at implementation time — the AI SDK evolves quickly and memorized APIs are unreliable.
- **Monorepo:** pnpm workspaces + Turborepo.
- **Demo app:** Next.js 16 (App Router, Node.js runtime via Fluid Compute).
- **Testing:** Vitest for unit tests (capture, redaction, prompt building, tool schemas). Playwright for the demo app end-to-end. `size-limit` in CI for the client bundle.
- **Bundle budget:** <35KB gzipped for the IIFE build.
- **License:** MIT.

### Why `docs/design/` specifically

- **Versioned with code.** Design decisions live in git, reviewed via PR, evolve visibly over time.
- **Dated filenames** (`YYYY-MM-DD-topic-design.md`) make chronology obvious and prevent naming collisions as the project grows.
- **Separate from `docs/api/` and `docs/getting-started.md`**, which serve end-users. `docs/design/` is for maintainers and contributors who want to understand *why*.
- **`docs/design/README.md` is an index** — listing all design docs with their status (draft / accepted / superseded / deprecated).
- Follow-up RFCs (v2 agent actions, Set-of-Mark pointing, hosted service, etc.) each land as new files here.

### Naming note: "pip" vs `@pip-help/core`

The product/brand name is **pip**. The npm package `pip` is almost certainly taken (and collides with Python's package manager for search). Packages are published under the `@pip-help/` scope: `@pip-help/core` for the widget, `@pip-help/server` for the backend adapter. Same pattern Vercel uses with `next` (package) vs "Next.js" (brand). Final scope name is a detail we can adjust once we check npm availability.

## Integration UX

**Vanilla HTML site (the one-liner):**

```html
<script
  src="https://unpkg.com/@pip-help/core"
  data-pip-endpoint="/api/pip"
></script>
```

**Next.js App Router:**

```tsx
// app/layout.tsx
import '@pip-help/core/auto';  // auto-mounts on document ready

// app/api/pip/route.ts
import { createPipHandler } from '@pip-help/server/next';
import { readFileSync } from 'node:fs';

// Uses Vercel AI Gateway by default — OIDC auth in production,
// AI_GATEWAY_API_KEY for local dev. No provider SDK needed.
export const POST = createPipHandler({
  model: 'anthropic/claude-sonnet-4.6',
  getContext: async () => readFileSync('./pip.md', 'utf-8'),
});
```

**Advanced (custom context):**

```ts
export const POST = createPipHandler({
  model: 'anthropic/claude-sonnet-4.6',
  getContext: async ({ query, url }) => {
    const chunks = await myVectorStore.search(query, { topK: 5 });
    return chunks.map(c => c.content).join('\n\n');
  },
});
```

Devs who want to bypass the Gateway (self-hosted, enterprise, air-gapped) can pass a direct provider model instance instead of the Gateway string — see `docs/api/server-config.md` for the escape hatch.

## Verification plan

1. **Demo app smoke test** — `pnpm dev` in `apps/demo`, open the page, click the mouse, ask "how do I create a new project?", verify the response streams and the overlay points at the right button.
2. **Playwright E2E** — automate the above: launch the demo, trigger the widget, assert a `highlight` tool call produces a visible SVG arrow with the correct bounding box.
3. **Unit tests (Vitest):**
   - `capture/`: screenshot includes visible content, excludes password fields, respects `[data-pip="redact"]` and `config.redact`.
   - `overlay/`: arrow positions correctly; dismisses on scroll/resize/click.
   - `transport/`: data-stream parser handles partial frames, tool calls, errors.
   - `server/prompt.ts`: system prompt includes context, history is trimmed to budget, tool schema matches the client parser.
4. **Bundle size gate** — `size-limit` CI check fails the build if the IIFE bundle exceeds 35KB gzipped.
5. **Manual privacy audit** — DevTools network tab; verify password fields and redacted selectors are blacked out in the outgoing screenshot and stripped from the DOM snapshot.
6. **Cross-browser spot check** — Chrome, Safari, Firefox. Shadow DOM + SVG overlay should work everywhere but worth verifying on a static HTML page and the Next.js demo.

## Open questions / known tradeoffs

- **Screenshot library choice** — `html-to-image` vs `modern-screenshot` vs `html2canvas`. Decision deferred to implementation; pick based on bundle size, cross-origin image behavior, and capture fidelity on the demo app.
- **Final npm scope name** — `@pip-help/*` is a placeholder; verify availability and adjust before first publish.
- **Coordinate drift** — even with scroll freeze, fonts loading or images lazy-loading between screenshot-time and overlay-render-time can shift layout. Mitigation: freeze as early as possible (on send, not on response). Accept as a known limitation; Set-of-Mark upgrade in v2 eliminates it.
- **Streaming protocol stability** — AI SDK v6's data stream protocol is the source of truth. Pin a minor version in `@pip-help/server` and re-test on upgrades.
- **Non-LLM tool execution** — v1 only emits `highlight` tool calls (read-only). When v2 adds `click`/`type`/`navigate`, the client needs a confirmation UX and a per-tool kill switch. Noted here so the v2 design doc can build on it.

## Roadmap after v1

These are not v1 commitments, but shape the v1 architecture so we don't paint ourselves into a corner:

- **v1.1** — Set-of-Mark pointing (numbered overlay on interactive elements) as a stability upgrade to coordinate-based pointing.
- **v1.2** — Express/Hono/Cloudflare adapters (thin wrappers over the existing generic handler).
- **v2** — Agent actions: click, type, navigate on the user's behalf, with per-action confirmation UX.
- **v2.x** — Optional hosted backend service for zero-config integration.
- **v3** — Vector-search reference implementation, published as `@pip-help/context-docs` (opt-in), so devs who want RAG out of the box can install one more package instead of writing `getContext()`.
